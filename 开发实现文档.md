### 2.2 Agent通信协议设计（MCP + A2A）

本系统采用双协议架构实现Agent间的标准化通信：

**MCP（Model Context Protocol）—— Agent与工具的交互**

| MCP原语 | 系统中的映射 | 说明 |
|:---|:---|:---|
| **Tools** | 星火大模型调用、向量检索、视频生成、TTS合成 | Agent可调用的函数，带结构化输入/输出 |
| **Resources** | 知识库文档、用户画像、学习记录 | Agent可读取的数据源 |
| **Prompts** | 画像分析提示词、内容生成提示词 | 可复用的提示词模板 |

**A2A（Agent-to-Agent Protocol）—— Agent间的协作**

| A2A概念 | 系统中的映射 | 说明 |
|:---|:---|:---|
| **Agent Card** | 每个智能体的能力声明文件 | 描述技能、认证方式、端点URL |
| **Task Lifecycle** | 任务状态：submitted→working→input-required→completed | 跨智能体的任务流转 |
| **Messages & Parts** | 智能体间传递的结构化消息 | 支持文本、文件、JSON数据 |

## 三、功能一：对话式学习画像自主构建

### 3.1 功能目标

用户通过自然对话，系统自动从对话中提取特征，构建包含至少6个维度的学习画像。采用 **ReAct（Reasoning + Acting）** 推理模式，画像随用户学习行为动态更新，支持跨会话持久化记忆。

### 3.2 六维度画像定义

| 维度 | 说明 | 取值示例 | 置信度 |
|:---|:---|:---|:---|
| **知识基础** | 对当前学习领域的掌握程度 | 初学/入门/进阶/精通 | 0.0-1.0 |
| **认知风格** | 偏好的信息接收方式 | 视觉型/听觉型/动觉型/阅读型 | 0.0-1.0 |
| **易错点偏好** | 常犯的错误类型或知识盲区 | 概念混淆/计算粗心/逻辑不清/记忆困难 | 0.0-1.0 |
| **学习偏好** | 喜欢的学习材料形式 | 视频为主/文字为主/练习为主/混合型 | 0.0-1.0 |
| **学习节奏** | 学习速度和耐力 | 快节奏/适中/慢节奏/碎片化 | 0.0-1.0 |
| **目标导向** | 学习目的 | 应试取证/技能实践/兴趣探索/学术研究 | 0.0-1.0 |

### 3.4 分步实现方案

#### 步骤1：MongoDB数据库设计

| 集合名称 | 存储内容 | 关键字段 |
|:---|:---|:---|
| users | 用户基本信息 | _id、用户名、创建时间、最后活跃时间 |
| profiles | 用户学习画像（版本化存储） | user_id、各维度值及置信度、版本号、更新时间 |
| profile_logs | 画像更新日志 | user_id、对话内容、分析结果、触发节点、LangGraph trace_id |
| learning_behaviors | 学习行为记录 | user_id、行为类型、时间戳、关联资源、行为详情 |

#### 步骤2：Letta Agent记忆系统集成

**三层记忆架构设计：**

| 记忆层级 | 对应Letta概念 | 存储内容 | 生命周期 |
|:---|:---|:---|:---|
| **核心记忆** | Core Memory | 当前活跃画像、会话上下文、当前学习目标 | 持续存在于上下文窗口 |
| **回忆记忆** | Recall Memory | 对话历史、学习行为时间线、画像演变记录 | 按时间索引，可检索 |
| **归档记忆** | Archival Memory | 历史画像版本、已完成的资源、学习成果 | 持久化存储，语义检索 |

**核心记忆操作示例：**

```python
# Letta Agent 自编辑记忆
agent.core_memory_append(
    label="学习偏好",
    content="用户在2024-03对话中明确表示偏好视频学习，视觉型认知风格"
)
agent.archival_memory_insert(
    content="用户在学习'神经网络基础'时，习题正确率仅40%，标记为知识薄弱点",
    metadata={"topic": "神经网络", "score": 0.4, "date": "2024-03-15"}
)
```

#### 步骤3：LangGraph画像分析工作流构建（ReAct模式）

**ReAct推理循环：**

```
Thought: 用户提到"我是计算机专业大二学生，准备考研408"
Action: 提取关键信息 → 专业=计算机、学历=本科、目标=考研408
Observation: 提取成功，知识基础维度信息不足
Thought: 需要进一步了解用户对数据结构的掌握程度
Action: 生成追问问题
```

**状态图流程：**

```
START → 对话预处理 → 画像分析(ReAct循环) → 置信度评估
                                                ↓
                                ┌───────────────┴───────────────┐
                                ▼                               ▼
                          置信度>0.6                      置信度≤0.6
                                │                               │
                                ▼                               ▼
                          增量合并 → Letta写入 → END       触发追问 → END
```

**LangGraph状态结构定义：**

| 状态字段 | 类型 | 说明 |
|:---|:---|:---|
| messages | list[BaseMessage] | 对话历史 |
| current_profile | dict | 当前画像快照 |
| analysis_result | dict | 6维度分析结果JSON |
| confidence_scores | dict | 各维度置信度分数 |
| letta_context | dict | Letta记忆检索结果 |
| reasoning_trace | list | ReAct推理链记录 |
| iteration_count | int | 当前推理循环次数 |
| langfuse_trace_id | str | Langfuse追踪ID |

#### 步骤4：后端接口设计

| 接口路径 | 功能 | 说明 |
|:---|:---|:---|
| POST /api/chat | 发送消息 | 接收用户消息，SSE流式返回AI回复+画像更新事件 |
| GET /api/profile | 获取画像 | 返回当前用户最新画像及各维度置信度 |
| PUT /api/profile | 手动修正画像 | 用户主动调整画像维度，触发工作流更新 |
| POST /api/behavior | 上报学习行为 | 记录行为数据，触发画像增量更新 |
| GET /api/profile/history | 画像历史 | 返回画像版本演变记录 |

#### 步骤5：画像增量更新策略

| 更新机制 | 规则说明 |
|:---|:---|
| 触发条件 | 每累计3条用户消息、完成练习、观看视频完成 |
| 合并规则 | 新维度值置信度>0.7时覆盖，0.5-0.7时标记待确认 |
| 冲突处理 | 以用户最新明确表达为准，Letta归档记忆保留历史 |
| 版本记录 | 每次更新创建新文档，profile_logs记录LangGraph trace |

#### 步骤6：前端画像展示与交互

| 前端组件 | 功能 |
|:---|:---|
| ECharts雷达图 | 展示6维度画像，数值映射为0-1 |
| 置信度标识 | 低置信度维度显示"推测中"标签，点击显示判断依据 |
| 手动修正弹窗 | 用户可点击任一维度手动修改 |
| 解释面板 | 展示每个维度的判断依据来源（来自哪轮对话） |

## 四、功能二：多智能体协同的资源生成

### 4.1 功能目标

系统根据用户需求和画像，由多智能体协作生成至少6种类型的个性化学习资源。采用 **Supervisor（主管）模式** 编排，支持 **并行执行、质量自检（Self-Refine）、Human-in-the-Loop审核**，资源基于 **Agentic RAG** 检索知识库确保事实准确性。

### 4.2 智能体角色定义（LangGraph节点 + MCP Tools）

| 智能体名称 | 核心职责 | 输出类型 | MCP Tool | 依赖关系 |
|:---|:---|:---|:---|:---|
| **主管智能体** | 理解意图、拆解任务、路由调度 | 任务分配指令 | - | 无 |
| **RAG检索智能体** | 基于用户需求检索知识库 | 相关文档片段 | `knowledge_retrieval` | 无 |
| **资料生成智能体** | 生成专业课程讲解文档 | Markdown文档 | `document_generation` | RAG检索智能体 |
| **思维导图智能体** | 将知识点转化为树状结构 | Markmap/Mermaid语法 | `mindmap_generation` | 资料生成智能体 |
| **习题生成智能体** | 生成不同难度的练习题 | 结构化JSON题目 | `exercise_generation` | 资料生成智能体 |
| **PPT生成智能体** | 生成教学课件 | python-pptx对象 | `ppt_generation` | 资料生成智能体 |
| **代码案例智能体** | 生成可运行的实操代码 | 代码+注释+说明 | `code_generation` | 资料生成智能体 |
| **视频生成智能体** | 生成多模态教学视频 | 视频URL/动画 | `video_generation` | 资料生成智能体 |
| **TTS语音智能体** | 生成知识点音频讲解 | 音频URL | `tts_generation` | 资料生成智能体 |
| **质量检查智能体** | Self-Refine自检+内容安全 | 质量评分+改进建议 | `quality_check` | 各生成智能体 |
| **结果聚合智能体** | 汇总、去重、格式化 | 最终资源列表 | - | 所有生成智能体 |

### 4.4 Agentic RAG 知识检索模块

**核心思路：** RAG不再是静态的"检索→生成"管线，而是Agent主动决策的检索行为。

```
用户需求："讲解神经网络的反向传播算法"
         │
         ▼
┌─────────────────────────┐
│ Agentic RAG 检索决策     │
│                         │
│ Thought: 这是一个概念   │
│ 解释型问题，需要检索    │
│ 教材中反向传播章节      │
│                         │
│ Action: 查询向量数据库  │
│ query="反向传播算法"    │
│ + BM25混合检索          │
└───────────┬─────────────┘
            │ Top-K文档片段
            ▼
┌─────────────────────────┐
│ BGE-Reranker 重排序     │
│ 语义相似度重排序         │
│ 选取最相关的3个片段      │
└───────────┬─────────────┘
            │
            ▼
┌─────────────────────────┐
│ 自我评估（CRAG模式）     │
│ 检索置信度 > 阈值？      │
│ Yes → 使用检索结果       │
│ No  → 调整Query重检索   │
└───────────┬─────────────┘
            │
            ▼
    检索结果注入各生成Agent的Prompt上下文
```

**知识库构建流程：**

```

  → LlamaParse文档解析（保留表格/公式）
  → 语义分块(chunk_size=512, overlap=64)
  → BGE-M3 Embedding向量化
  → 存入MongoDB Atlas Vector Search
  → BM25倒排索引（混合检索）
```

**GraphRAG增强：**

from pydantic import BaseModel, Field
from typing import List
from enum import Enum

class QuestionType(str, Enum):
    MULTIPLE_CHOICE = "multiple_choice"
    FILL_BLANK = "fill_blank"
    SHORT_ANSWER = "short_answer"
    TRUE_FALSE = "true_false"

class Exercise(BaseModel):
    """单道练习题的结构化输出"""
    question_type: QuestionType
    difficulty: int = Field(ge=1, le=5, description="难度等级1-5")
    question: str = Field(description="题目内容")
    options: List[str] | None = Field(default=None, description="选择题选项")
    answer: str = Field(description="正确答案")
    explanation: str = Field(description="答案解析，引用知识来源")
    knowledge_source: str = Field(description="对应知识点，引用自哪篇文档")

class ExerciseSet(BaseModel):
    """习题集的结构化输出"""
    topic: str
    exercises: List[Exercise] = Field(min_length=5, max_length=10)
    rag_sources: List[str] = Field(description="RAG检索到的参考文档列表")
```

│                                                                 │
│  Layer 4: NeMo Guardrails轨道                                  │
│  ├─ 学术内容轨道：确保内容学术规范                              │
│  ├─ 安全轨道：拦截违规内容                                      │
│  └─ 越狱防御：检测Prompt注入攻击                               │
│                                                                 │
│  Layer 5: Human-in-the-Loop（可选）                             │
│  ├─ 关键资源生成后展示给用户确认                                │
│  └─ 用户可标记"不准确"触发重新生成                             │
└─────────────────────────────────────────────────────────────────┘
```

### 6.2 事实性校验实现

| 校验维度 | 实现方式 | 触发条件 |
|:---|:---|:---|
| 语义相似度 | 生成内容与RAG检索片段做余弦相似度比对 | 每次生成后 |
| 原子事实分解 | 将生成内容拆解为原子事实，逐条验证 | 质量检查阶段 |
| 引用溯源 | 每段内容标注"[来源: 文档名, 页码]" | 生成时自动添加 |
| LLM-as-Judge | 第二个LLM实例验证第一个的输出 | 关键内容生成 |

## 八、技术选型依据与创新点总结

### 8.1 技术选型依据

| 技术 | 选型理由 | 替代方案对比 |
|:---|:---|:---|
| LangGraph | 支持循环图、持久化检查点、Human-in-the-Loop，多智能体编排首选 | CrewAI（角色更固定）、AutoGen（事件驱动但学习曲线高） |
| MCP | Anthropic开源标准，统一Agent-Tool交互，生态丰富 | 自定义API（无标准，难扩展） |
| A2A | Google开源标准，支持跨框架Agent通信，50+合作伙伴 | LangChain Hub（仅限LangChain生态） |
| Letta | 唯一支持Agent自编辑记忆的框架，三层记忆架构 | Redis缓存（无语义检索）、MongoDB（需自建记忆逻辑） |
| instructor | 强制Pydantic Schema，自动重试，支持多LLM提供商 | LangChain OutputParser（重试机制弱） |
| Guardrails AI | RAIL规范+Python验证器，防幻觉和内容安全一站式 | 自建过滤（开发量大） |
| MongoDB Atlas Vector Search | 文档存储+向量检索一体化，无需额外向量数据库 | 单独部署Milvus/Qdrant（运维成本高） |
| Langfuse | 开源可观测性，自部署无数据外泄，支持Agent追踪 | LangSmith（SaaS，数据外泄风险） |
| CosyVoice 2 | 阿里开源，零样本语音克隆，中文效果最佳 | Edge TTS（免费但无克隆能力） |
| E2B | 云端沙箱代码执行，安全隔离，开源SDK | 本地Jupyter（安全风险高） |

### 8.2 创新点总结

| 创新点 | 描述 | 赛题评分关联 |
|:---|:---|:---|
| **1. MCP+A2A双协议Agent架构** | 采用2025年最新开源标准，实现Agent-Tool和Agent-Agent的标准化通信，具备可扩展性和互操作性 | 创新价值(35%) |
| **2. Agentic RAG + GraphRAG混合检索** | RAG不再是静态管线，而是Agent主动决策的检索行为，结合知识图谱多跳推理 | 创新价值(35%) + 功能实现(45%) |
| **3. Letta三层记忆持久化** | Agent具备跨会话的自编辑记忆能力，核心/回忆/归档三层架构模拟人类记忆 | 功能实现(45%) |
| **4. instructor结构化输出** | Pydantic Schema强制约束LLM输出，自动重试保障数据一致性 | 功能实现(45%) |
| **5. Self-Refine + Guardrails多层防幻觉** | 五层防护架构（RAG锚定→结构化输出→Guardrails验证→NeMo轨道→人工确认） | 功能实现(45%) |
| **6. Plan-and-Execute路径规划** | 先全局规划再逐步执行，支持反馈驱动的动态调整，比纯ReAct更适合长期学习规划 | 创新价值(35%) |
| **7. 多模态资源生成矩阵** | 覆盖文本/图片/思维导图/PPT/代码/视频/音频七种模态，非单一视频生成 | 功能实现(45%) |
| **8. Langfuse全链路可观测** | Agent执行过程可追溯、可评估、可优化，支撑持续迭代 | 文档丰富度(10%) |
