## 二、技术选型决策记录

### 2.1 为什么选 LangGraph 而不是 CrewAI / AutoGen

| 对比维度 | LangGraph | CrewAI | AutoGen |
|:---|:---|:---|:---|
| 循环支持 | 原生支持有向循环图 | 有限 | 支持但架构复杂 |
| 持久化检查点 | 内置 | 无 | 需额外配置 |
| Human-in-the-Loop | 内置interrupt机制 | 有限 | 需手动实现 |
| 子图组合 | 支持嵌套子图 | 不支持 | 不支持 |
| 社区生态 | 最活跃，更新最快 | 中等 | 分裂（AG2 fork） |
| 可观测性 | LangSmith/Langfuse原生集成 | 有限 | 需额外工具 |

**结论：** LangGraph 在状态管理、循环、检查点方面最适合本系统的三类Agent工作流。

### 2.2 为什么用 litellm 做 LLM 抽象

- 一行代码切换 Provider：`model="openai/gpt-4o"` → `model="deepseek/deepseek-chat"` → `model="spark/v4.0"`
- 统一的 `completion()` 接口，所有 Agent 代码无需改动
- 内置重试、超时、流式支持
- 支持 100+ 模型提供商

### 2.3 为什么不用 Docker

- 本地已有 MongoDB 8.3，直接 `mongod` 启动即可
- 开发阶段无需容器化，减少复杂度
- 比赛提交时可在文档中说明"需安装MongoDB"即可

### 2.4 向量检索方案选择

| 方案 | 优点 | 缺点 | 选择 |
|:---|:---|:---|:---|
| MongoDB Atlas Vector Search | 与现有MongoDB一体 | 需Atlas云服务，本地不支持 | ❌ |
| 单独部署 Milvus/Qdrant | 性能最强 | 需额外安装，运维成本高 | ❌ |
| **简易余弦相似度** | 零依赖，用numpy实现 | 大数据量性能差 | ✅ 选用 |
| ChromaDB | 轻量嵌入式 | 额外依赖 | 备选 |

**理由：** 比赛场景知识库规模有限（1-3门课程），numpy余弦相似度完全够用，零额外依赖。

## 三、核心数据流设计

### 3.1 功能一：对话式画像构建数据流

```
用户输入消息
    │
    ▼
[FastAPI 接收 POST /api/chat]
    │
    ├──→ 存储消息到 MongoDB chat_sessions
    │
    ├──→ 启动 LangGraph ProfileAgent 工作流
    │        │
    │        ├── Node 1: 对话预处理
    │        │   提取关键信息（专业、水平、偏好）
    │        │
    │        ├── Node 2: 画像分析（调用LLM）
    │        │   输出6维度分析 JSON
    │        │
    │        ├── Node 3: 置信度评估
    │        │   每个维度计算0-1置信度
    │        │
    │        ├── 条件边: 置信度 > 0.6?
    │        │   Yes → Node 4: 增量合并到现有画像
    │        │   No  → Node 5: 生成追问问题
    │        │
    │        └── Node 6: 写入记忆系统
    │            写入 Letta三层记忆
    │
    ├──→ 返回 AI 回复（SSE流式）
    │
    └──→ 返回画像更新事件（前端实时更新雷达图）
```

### 3.2 功能二：多智能体资源生成数据流

```
用户输入学习需求（如"讲解神经网络反向传播"）
    │
    ▼
[FastAPI 接收 POST /api/generate]
    │
    ├──→ 返回 task_id（前端用于轮询进度）
    │
    └──→ 异步启动 LangGraph ResourceAgent 工作流
             │
             ├── Supervisor Node: 任务拆解
             │   分析需求 → 决定生成哪些资源（至少6种）
             │
             ├── RAG 检索 Node:
             │   查询知识库 → 返回相关文档片段
             │   注入到后续所有Agent的上下文中
             │
             ├── 资料生成 Node（串行，最先执行）:
             │   基于RAG结果生成Markdown讲解文档
             │
             ├── 并行生成（4个Agent同时执行）:
             │   ├── 思维导图Agent → Markmap语法
             │   ├── 习题Agent → 结构化JSON（instructor验证）
             │   ├── PPT Agent → python-pptx大纲
             │   ├── 代码Agent → 可运行代码示例
             │   ├── 视频Agent → 视频脚本+URL
             │   └── TTS Agent → 音频讲解URL
             │
             ├── 质量检查 Node（Self-Refine）:
             │   自动评分 → ≥80分通过 → <80分重新生成（最多3次）
             │
             ├── 安全过滤 Node（Guardrails）:
             │   事实性校验 + 敏感词检测
             │
             └── 结果聚合 Node:
                 去重、格式化、存储到MongoDB
                 更新任务状态为 completed
```

### 3.3 功能三：学习路径规划数据流

```
用户触发路径规划（选择主题）
    │
    ▼
[FastAPI 接收 POST /api/path/plan]
    │
    └──→ 启动 LangGraph PathAgent 工作流
             │
             ├── Planner Node:
             │   输入：用户画像 + 学习目标 + 知识库结构
             │   输出：5-8个学习步骤（含难度、时长、资源类型）
             │
             ├── Executor Node（逐步执行）:
             │   对每个步骤：
             │     ├── 查询MongoDB已有资源 → 命中则直接推送
             │     └── 未命中 → 触发功能二的资源生成
             │
             ├── 进度追踪 Node:
             │   记录每步完成状态，更新Letta记忆
             │
             └── 反馈处理（条件循环）:
                 用户反馈"太难" → 插入前置知识步骤 → 重新规划
                 用户反馈"太简单" → 跳过/合并步骤 → 重新规划
                 循环上限：最多3次重新规划
```

## 四、数据库 Schema 设计

### 4.1 MongoDB 集合总览

| 集合名称 | 用途 | 预估文档量 |
|:---|:---|:---|
| users | 用户基本信息 | 按用户数 |
| profiles | 用户学习画像（版本化） | 用户数 × 平均10版本 |
| profile_logs | 画像更新日志 | 每次对话触发 |
| chat_sessions | 对话会话 | 每用户多个会话 |
| chat_messages | 对话消息 | 大量 |
| resources | 生成的学习资源 | 大量 |
| learning_paths | 学习路径 | 每用户多条 |
| learning_progress | 学习进度 | 每路径×步骤数 |
| tasks | 异步任务状态 | 并发任务数 |
| knowledge_docs | 知识库文档（含向量） | 课程文档切片 |

## 六、Agent 工作流详细设计

### 6.1 ProfileAgent（画像分析 - ReAct模式）

**状态结构（StateGraph State）：**
```
State {
  messages: List[ChatMessage]     # 对话历史
  current_profile: Dict           # 当前画像
  analysis_result: Dict           # 分析结果
  confidence_scores: Dict         # 各维度置信度
  reasoning_trace: List[str]      # ReAct推理链
  iteration: Int                  # 当前循环次数
  max_iterations: Int = 5         # 最大循环次数
}
```

**节点定义：**

| 节点名 | 类型 | 输入 | 输出 | 调用LLM |
|:---|:---|:---|:---|:---|
| preprocess | 计算节点 | messages | filtered_messages | 否 |
| analyze | LLM节点 | filtered_messages + current_profile | analysis_result | 是 |
| evaluate_confidence | 计算节点 | analysis_result | confidence_scores | 否 |
| merge_profile | 计算节点 | analysis_result + current_profile | updated_profile | 否 |
| generate_followup | LLM节点 | low_confidence_dimensions | followup_question | 是 |
| write_memory | 计算节点 | updated_profile | memory_written | 否 |

**条件边逻辑：**
```
evaluate_confidence → 
  if any(confidence < 0.6) → generate_followup → END
  else → merge_profile → write_memory → END
```

**循环控制：** ReAct循环最多5次，每次循环尝试从对话中提取更多信息。

### 6.2 ResourceAgent（资源生成 - Supervisor模式）

**状态结构：**
```
State {
  user_id: str
  topic: str
  user_profile: Dict
  rag_results: List[Document]     # RAG检索结果
  material_document: str          # 资料员输出
  mindmap: Dict                   # 导图输出
  exercises: List[Dict]           # 习题输出
  ppt_outline: Dict               # PPT大纲输出
  code_example: Dict              # 代码输出
  video_info: Dict                # 视频输出
  audio_info: Dict                # 音频输出
  quality_report: Dict            # 质量检查报告
  generated_resources: List[Dict] # 最终资源列表
  current_step: str
  errors: List[str]
  task_id: str
}
```

**执行顺序：**
```
Supervisor → RAG检索 → 资料生成 → [导图||习题||PPT||代码||视频||TTS] → 质量检查 → 安全过滤 → 结果聚合
```

**并行执行实现：** LangGraph的 `Send` API 或 Python `asyncio.gather` 在工作流内实现并行。

**质量检查循环：**
```
quality_check → 
  if score >= 80 → safety_filter → aggregate → END
  if score < 80 and retries < 3 → 重新触发对应Agent → quality_check
  if score < 80 and retries >= 3 → 标记为"低质量"但仍输出 → aggregate → END
```

### 6.3 PathAgent（路径规划 - Plan-and-Execute模式）

**状态结构：**
```
State {
  user_id: str
  topic: str
  user_profile: Dict
  plan_steps: List[Step]          # 规划的学习步骤
  current_step_index: Int         # 当前执行到第几步
  completed_steps: List[str]      # 已完成的步骤ID
  feedback: str                   # 用户反馈
  iteration_count: Int            # 重新规划次数
  max_iterations: Int = 3
  final_path: LearningPath        # 最终路径
}
```

**Planner节点输出格式：**
```json
{
  "steps": [
    {
      "step_id": "step_1",
      "name": "神经网络基础概念",
      "objective": "理解神经元、激活函数、前向传播",
      "difficulty": "基础",
      "duration_minutes": 30,
      "resource_types": ["document", "mindmap", "exercise"],
      "prerequisites": [],
      "rag_query": "神经网络基础 感知机 激活函数"
    }
  ]
}
```

**反馈处理分支：**
```
feedback ==
  "too_hard"   → 插入1-2个前置步骤 → iteration_count++ → 重新Planner
  "too_easy"   → 跳过当前+合并后续 → iteration_count++ → 重新Planner
  "need_help"  → 触发答疑Agent → 不修改路径
  "completed"  → current_step_index++ → 进入下一步
```

## 八、服务层设计

### 8.1 LLM 服务（services/llm.py）

**职责：** 统一封装所有LLM调用，屏蔽Provider差异。

**核心接口：**
- `chat(model, messages, temperature, max_tokens) → str` — 同步调用
- `chat_stream(model, messages, temperature, max_tokens) → AsyncIterator[str]` — 流式调用
- `structured_output(model, messages, response_model) → BaseModel` — 结构化输出（instructor）

**配置（通过.env）：**
```
LLM_PROVIDER=openai        # 或 deepseek, spark等
LLM_API_KEY=sk-xxx
LLM_MODEL=gpt-4o
LLM_BASE_URL=              # 可选，自定义API地址
```

### 8.2 RAG 服务（services/rag.py）

**职责：** 知识库检索，支持混合检索（向量+关键词）。

**核心接口：**
- `index_document(course_id, chunks)` — 索引文档切片
- `retrieve(query, top_k=5) → List[Document语义检索 — 向量相似度
- `hybrid_retrieve(query, top_k=5) → List[Document]` — 混合检索（向量+BM25）

**检索流程：**
```
query → BGE-M3 Embedding → 余弦相似度排序 → Top-K → （可选）重排序 → 返回
```

**向量化方案：**
- 使用 sentence-transformers 加载 BGE-M3 模型
- 768维向量，存入MongoDB的knowledge_docs集合
- 查询时用numpy计算余弦相似度

### 8.3 记忆服务（services/memory.py）

**职责：** 管理Agent的三层记忆，模拟Letta架构。

**三层记忆实现：**

| 层级 | 存储位置 | 读写方式 | 用途 |
|:---|:---|:---|:---|
| 核心记忆 | MongoDB + 上下文注入 | Agent每次调用自动注入 | 当前活跃画像、会话目标 |
| 回忆记忆 | MongoDB chat_messages | 按时间范围查询 | 对话历史、行为时间线 |
| 归档记忆 | MongoDB + 向量索引 | 语义检索 | 历史画像、学习成果、知识薄弱点 |

**核心接口：**
- `get_core_memory(user_id) → Dict` — 获取核心记忆（注入到Agent上下文）
- `update_core_memory(user_id, key, value)` — 更新核心记忆
- `search_recall_memory(user_id, query, limit) → List` — 搜索回忆记忆
- `archive_memory(user_id, content, metadata)` — 写入归档记忆
- `search_archival_memory(user_id, query, limit) → List` — 语义搜索归档记忆

## 九、Agent工具设计（MCP风格）

### 9.1 工具定义格式

每个工具遵循MCP规范，包含：名称、描述、输入Schema、输出Schema。

#### knowledge_search 工具
```
名称: knowledge_search
描述: 在课程知识库中检索与查询相关的文档片段
输入: { query: str, top_k: int = 5, course_id: str = null }
输出: { documents: [{content, source, score}], total_results: int }
```

#### code_executor 工具（可选）
```
名称: code_executor
描述: 在安全沙箱中执行Python代码并返回结果
输入: { code: str, language: str = "python" }
输出: { output: str, error: str|null, execution_time_ms: int }
```

### 9.2 工具注册到Agent

LangGraph中通过 `ToolNode` 将工具注册到工作流：
- 工具作为Agent的可调用函数
- Agent在ReAct循环中自主决定何时调用哪个工具
- 工具调用结果反馈到Agent的Observation中

## 十二、风险与应对

| 风险 | 影响 | 应对方案 |
|:---|:---|:---|
| LLM API 调用超时 | 生成卡住 | 设置30秒超时 + 重试3次 + 前端显示"请求超时，请重试" |
| LLM 输出格式不规范 | 结构化解析失败 | instructor自动重试 + 兜底正则解析 |
| MongoDB 连接失败 | 数据读写失败 | 连接池 + 重试机制 + 优雅降级 |
| 知识库检索无结果 | RAG返回空 | 回退到LLM自身知识 + 提示"未找到相关内容" |
| 并行生成资源冲突 | 竞态条件 | MongoDB事务 + 任务队列串行化 |
| 前端SSE断连 | 进度丢失 | 自动重连 + 任务状态持久化在MongoDB |
